# 3.8. Метадані — конкретні приклади з живого API

> Дані витягнуто з `https://data.gov.ua/api/3/action/*` (CKAN production), вибірка 30 датасетів з `package_search?q=*:*&rows=30`, плюс `group_list`, `organization_list`. Дата збору: 2026-05-25.

> **Бічна знахідка:** канонічно працює база `https://data.gov.ua/api/3/action/`, тоді як офіційний snippet документує `https://opendata.gov.ua/uk_UA/api/3/action/`. Два домени не повністю взаємозамінні (різна reachability, можлива різна політика CORS/throttling). Дублювання та брак канонічного hostname — окрема проблема `3.0`.

| `podatky` | Державні доходи і видатки | slug не відповідає title (фактично "податки", а title про «доходи і видатки») — **семантичний дрейф** |
| `ekonomika` | Економіка та бізнес | |

| `ekolohiia` | Навколишнє середовище | `kh` для **х**? Тут немає Х. Slug = "екологія", title про «навколишнє середовище» — **семантичний дрейф** |

## C. Resource-файли: 6% — MIXED (кирилиця + латиниця в одному рядку)

З 737 ресурсів у 30 датасетах:

```
{'uk': 585, 'en': 109, 'MIXED': 43, 'uuid': 13}
```

Приклади MIXED (укр. назва + латинське розширення/латинські вкраплення) (загалом, це ок, якщо це тільки розширення, чи щось таке дрібне):

```
Організаційна структура.xlsx
Структура на 01.01.2021.pdf
Організаційна структура на 09.03.2026.pdf
Річні зведені основні фінансові показники виконання фінансових планів підприємст(ств)
Додаток 2 до п. 1 розділу VІ Порядку проходження стажування прокурора- стажиста
Додаток 3 до п. 11 розділу VІІ Порядку проходження стажування прокурора- стажиста
```

> Окремо: `VІ` та `VІІ` — це **римські цифри написані ук-кирилицею** (Cyr `І` `U+0406`), а не латинською `I` `U+0049`. CKAN/Solr нормалізує по-різному → пошук `q=VII` не знайде, лише `q=VІІ`. Це **юнікодний кошмар** для майбутнього MCP-tool.

## D. Английські camelCase у файлах для повністю українських датасетів

Dataset `weaponswanted` (title: «Інформація про викрадену, втрачену зброю»):

| Resource name | format | mimetype |
|---|---|---|
| `WeaponsWanted.shema.json` | JSON | `application/json` |
| `WeaponsWanted.json` | JSON | `application/json` |

**Помилки:**
1. `shema.json` — друкарська помилка, має бути `schema.json`. Зберіглася в продакшні роками.
2. Немає поля `description` для ресурсу — користувач не розуміє, що `WeaponsWanted.shema.json` — це JSON Schema, а `WeaponsWanted.json` — дані.

Дублікат патерну: dataset `30b67898-...` («Викрадені мобільні»):

```
WantedMT.json
WantedMT_schema.json   ← тут правильно "schema"
```

Тобто **одна організація використовує дві різні конвенції** в межах одного типу датасету.

## E. Відсутність мультимовних полів (i18n)

`package_show` повертає:

```json
{
  "title": "Інформація про викрадену, втрачену зброю",
  "notes": "...",
  "license_id": "cc-by",
  "license_title": "Creative Commons Attribution"
}
```

**Немає:**
- `title_en` / `title_uk` — однієї назви на двох мовах.
- `notes_en` / `notes_uk`.
- `tags_en` / `tags_uk` — теги тільки українською (`['викрадена', 'зброя', 'розшук']`). Запит `?q=stolen+weapons` нічого не знайде.
- DCAT-AP/Dublin Core полів: `dct:language`, `dcat:keyword` per language.

Це **блокує** використання порталу AI-агентами, що працюють англійською (а більшість foundation-моделей оптимізовані саме під en).

## G. Liцензії — поки що єдині, але без машинно-читаної URI

У вибірці 30 датасетів **100% мають**:

```
license_id    = "cc-by"
license_title = "Creative Commons Attribution"
```

**Чого бракує:**
- `license_url` (наприклад, `https://creativecommons.org/licenses/by/4.0/`).
- Версії ліцензії (`4.0` / `3.0` / `2.5`)— критично для юридичної інтерпретації.
- DCAT-AP `dct:license` як URI.

(У ширшій вибірці, ймовірно, з'являться інші ліцензії — потрібна верифікація на повному каталозі.)

## I. Як MCP-сервер має це адресувати

1. **`search_datasets`** — після CKAN-виклику робити бек-пошук по transliterated-tags (n-gram на основі словника UA→EN). Запит «weapons» → внутрішньо `q=зброя OR weapons`.
2. **Locale-агностичний tag-індекс** — кешувати `tag_list`, будувати локальний UA↔EN мапінг (тонкий, з ~500 топ-тегів).
3. **Нормалізація UUID-slug'ів** — у відповіді MCP-tool завжди додавати `human_url` (з title) + `canonical_url` (з UUID).
4. **Unicode-нормалізація** — `q` параметр нормалізувати NFKC + замінювати кириличні «І/А/Е/Р/Х» що зовні виглядають як латинські, на обидва варіанти у Solr-запиті (`(VII OR VІІ)`).
5. **License enrichment** — мапінг `license_id` → `license_url` + `license_version` локально.
6. **Group-semantic mapping** — локальна таблиця: `podatky` → не показувати як «податки», а як «державні фінанси».

Ці шість пунктів треба додати до розділу **5.1.16 `summarize_resource`** та **6.1 Resources** у PLAN.md (post-task TODO).
